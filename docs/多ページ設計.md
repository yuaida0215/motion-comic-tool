# 多ページ（章まるごと）対応 設計

現状は「1ページ = 1プロジェクト = 1つのJSON」。実運用では**漫画を何ページも入れて1つの作品（章/エピソード）**として動画化したい。その器をどう拡張するかの設計（実装は後フェーズ）。

## 結論（推奨）
- **器＝A：多ページを1プロジェクト（=1章）に束ねる**。声の配役・背景テーマ・コマ通し番号を**全ページ共通**＝キャラの声も雰囲気も最後まで一貫。
- **レンダ＝ページ（orシーン）単位でクリップ書き出し → 任意で連結して1本**。20ページ一発レンダを避け、再書き出しも変更ページだけで済む。
- **納品＝両方**：個別クリップ（個別納品志向に合う）＋「つないで1本」（章通し視聴用）。
- B案（1ページ1プロジェクトのまま最後に連結だけ）は速いが、声/テーマ/編集がページごとにバラバラ＝一貫性が手作業になるので不採用。ただしBの「連結」発想はAのレンダ戦略に取り込む。

## 現状（拡張前）の要点
- `meta.source_image` = 1枚。`shots[]` = そのページのコマ。G0→G4 で1ページ→1mp4。
- 演出レイヤー（色背景B・額装・カードD/E・トランジションC・口パク①②）は **project / shot 単位で実装済み**。`voice_cast` と `meta.theme` は project 共通。
- timeline は `timeline.shots[i] ↔ project.shots[i]` の index 対応が g4(reveal/mouth)・MotionComic で前提（=この不変条件は多ページでも「ページ内」で守る）。

## データモデル変更
- `meta.pages: [{ id, source_image, index }]` を追加。**既存の `meta.source_image` は pages[0] の別名として維持**（後方互換）。1ページの既存プロジェクトは parse 時に `pages=[1件]` へ自動移行。
- `ShotSchema` に `page_id`（どのページのコマか）。`bbox` は**そのページ画像内の座標**（従来どおり）。shot の id は章で通し（`shot_01..`）でページ跨ぎ一意。
- `cards`（D/E）は既に project 直下＝**章頭タイトル / ページ境界アイキャッチ**に使える。配置に `after_page`（指定ページの後）を追加すると章構成が組める。
- `voice_cast` / `meta.theme` / `target_duration_sec` は project 共通のまま＝**全ページ一貫**（追加実装ほぼ不要）。
- `outputs`: `demo_mp4`（連結1本）＋ `page_clips: [{page_id, mp4}]`（個別）。
- `SCHEMA_VERSION` を上げる（追加のみ・移行コード不要、既定値で現状再現）。

## パイプライン変更
- **取り込み**：複数画像アップロード（or 既存プロジェクトに「ページ追加」）。各ページを storage `pages/<page_id>.png` に保存。
- **G0**：ページごとに検出 → shot に `page_id` 付与。
- **G1**：全ページのコマをまとめて章全体のコンテ（ビート/テーマ/トランジション/カード）を設計。声の配役は章共通。
- **G2**：`loadSourceImage` を**ページ指定**に拡張し、shot.page_id のページ画像から該当コマを文字消し。
- **G3**：章共通の `voice_cast` でTTS＝**同キャラ＝同声が全ページで保たれる**。
- **G4**：**ページ単位でクリップをレンダ**（そのページの shot の timeline）→ `pages/<id>.mp4`。さらに**連結**（ffmpeg concat）＋章頭/境界のカード・トランジション・**章通しBGM**を重ねて `episode.mp4`。

## レンダ戦略（ここが肝）
- 1ページ = 1クリップ（数コマ・短時間・**途中保存/再開可・並列可**）。**変更ページだけ再レンダ**＝コスト/時間を抑える。
- 連結は ffmpeg concat（再エンコード最小）。**BGMは連結後に章全体へ1本**敷く（ページごとに切れない）＝【決定】。ページ境界は**トランジション（白フラッシュ等）のみ**。**自動アイキャッチは入れない**（章頭タイトルや任意のアイキャッチは「文字カード」UIで手動挿入）＝【決定】。
- これで「20ページ一発レンダ」を避けつつ「1本」も出せる。timeline は**ページ境界を持つ episode timeline** に拡張するが、index 対応は「ページ内」で維持。

## UI 変更
- 新規作成：**複数ファイル選択**（or プロジェクトに「ページ追加」ボタン）。並び替え可。
- レビュー：ページのタブ/縦スクロールで切替、各ページに検出枠オーバーレイ。コマ一覧をページ見出しでグルーピング。
- 出力：**個別クリップ一覧 ＋「つないで1本」ボタン**。

## 後方互換・移行
- 既存1ページプロジェクト＝`pages` 長さ1に自動移行（parse で `meta.source_image` → `pages[0]`、`shot.page_id` 未設定は page0）。挙動不変。

## コスト/スケール
- 章20ページ ≈ 140コマ → G0/G1(vision)・G3(TTS) が嵩む。ページ単位処理＝途中保存/再開＋変更ページのみ再処理でコスト抑制。レンダもページ単位で短く。

## 段階ロードマップ（実装は後）

## 決定事項（2026-06-30 合田さん）
1. **ページ間の自動アイキャッチ：入れない**。ページ境界はトランジション（白フラッシュ等）のみ。章頭タイトル/任意アイキャッチは「文字カード」UIで手動挿入。
2. **BGM：章通し1本**（連結後に章全体へ）。
3. **想定ページ数：章スケール（〜10〜20ページ＋）**。レンダはページ単位なので枚数は問わない（変更ページのみ再レンダ・並列）。
4. **ページの並び：手動（合田さん主導）で確定**。既定はファイル名順、UIで並べ替え可。

## 要対応の実コード箇所（実装メモ）
※ 検証ワークフローはセッション上限で verify が中断（31件未検証）。下記は実コード知識ベースの要対応点。上限解除後に再検証する。
- **単一ページ前提を全部ページ対応に**：`lib/image.ts loadSourceImage` と呼び出し元（`lib/gates/g0.ts`/`g2.ts`/`g4.ts`/`lib/mouth.ts` detectMouths・generateOpenMouth）は `meta.source_image` 1枚を読む。shot.page_id のページ画像を読むAPI（例 `loadPageImage(project, page_id)`）に置換。**G4のreveal切り出し・口パク素材生成も該当ページ画像から行う**。
- **G1/G3の章スケール（page0しか見ない既知制限）**：現状 **G1（コンテ）と G3（話者/感情/声の割当）は1回のvision呼び出しで page0 画像しか見ない**まま全ページのコマを処理する（M0+M1では未対応）。20ページを1回に投げると入力過大/品質低下、かつ2ページ目以降に初登場する話者の声カテゴリ誤割当の恐れ → **ページ単位（or chunk）でG1/G3のvisionを回し、章共通方針（テーマ/声配役）だけ先に決める**設計にする（将来M）。当面は**承認画面の声配役UIで人が直せる**ので実害は限定的（クラッシュはしない）。
- **timeline/レンダ**：`computeTimeline` をページ境界つきに拡張。**index対応 `timeline.shots[i]↔project.shots[i]` はページ内で維持**。G4は**ページ単位でクリップ書き出し→ffmpeg concat**（ffmpeg-static 利用可）。BGMは連結後に1本。
- **後方互換**：`meta.source_image` 参照箇所（loadSourceImage・assetURL・Root.tsx）は pages[0] にフォールバック。parseで1ページ→pages長さ1へ移行。shot.page_id 既定=page0。
- **storage/UI**：source は `pages/<page_id>.png`、出力は `page_clips[]`＋連結 `episode.mp4`。アップロードは複数ファイル＋並べ替え。レビューはページ切替。localStorage下書きは safeParseProject 経由で pages/page_id を既定補完（既対応の仕組みで吸収）。
